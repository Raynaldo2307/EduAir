🧱 schools table – structure

Columns
	1.	id – INT, primary key, auto-increment
	•	Unique number for each school.
	2.	name – VARCHAR(100), NOT NULL
	•	School name. Example: "St. Catherine High".
	3.	parish – VARCHAR(100), NOT NULL
	•	Parish where the school is. Example: "St. Catherine", "Kingston".
	4.	school_type – ENUM('basic','primary','secondary','all_age','other'), NOT NULL, default 'primary'
	•	What kind of school:
	•	basic – basic / infant
	•	primary – primary
	•	secondary – high school
	•	all_age – all-age / primary & junior high
	•	other – anything else
	5.	is_shift_school – TINYINT(1), NOT NULL, default 0
	•	0 = normal whole-day school
	•	1 = shift school (morning/afternoon system)
	6.	default_shift_type – ENUM('morning','afternoon','whole_day'), NOT NULL, default 'whole_day'
	•	What kind of day the school mainly runs:
	•	'whole_day' – regular day (most schools)
	•	'morning' or 'afternoon' – if the school mostly uses that shift
	7.	latitude – DECIMAL(10,8)
	8.	longitude – DECIMAL(11,8)
	•	GPS location for geofencing (where the school is on the map).
	9.	radius_meters – INT
	•	How far from the school (in metres) a student can be and still “count” as on campus
(used by attendance geofence).
	10.	created_at – DATETIME, NOT NULL, default CURRENT_TIMESTAMP
	•	When this school row was created.
	11.	updated_at – DATETIME, NOT NULL, default CURRENT_TIMESTAMP and auto-updates
	•	Last time this school row was changed.

⸻

🔐 Index / uniqueness rules
	•	Primary key
	•	PRIMARY KEY (id)
	•	Unique constraint
	•	UNIQUE KEY uniq_school_name_parish (name, parish)
	•	Means you cannot have two rows with the same name + parish
(no duplicate "Papine High" / "Kingston").

1. Current students table (what I see in the screenshot)

Columns
	1.	id – INT, PRIMARY KEY, AUTO_INCREMENT
	•	Unique student row id.
	2.	school_id – INT NOT NULL
	•	Every student must belong to a school.
	•	Has a foreign key to schools(id).
	3.	first_name – VARCHAR(100) NOT NULL
	4.	last_name – VARCHAR(100) NOT NULL
	•	No more blank names slipping in.
	5.	phone_number – VARCHAR(20) (nullable)
	•	Optional contact number.
	6.	status – ENUM('active','inactive','graduated') NOT NULL DEFAULT 'active'
	•	Simple lifecycle:
	•	active – current student
	•	inactive – transferred/expelled/dropped out
	•	graduated – finished school
	7.	created_at – TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
	•	When the student row was created.
	8.	homeroom_class_id – INT (nullable)
	•	Optional link to their main class.
	•	Has a foreign key to classes(id).
	9.	updated_at – TIMESTAMP NOT NULL   DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
	•	Auto-updates every time you change the row.
	10.	user_id – INT, UNIQUE (nullable)

	•	0 or 1 app account per student.
	•	Foreign key to users(id).

⸻

2. Indexes & constraints

From your screenshots:
	•	Indexes
	•	PRIMARY KEY (id)
	•	Index on school_id
	•	Index on homeroom_class_id
	•	Unique index on user_id
	•	Foreign keys
	•	students.school_id  → schools.id
	•	students.homeroom_class_id → classes.id
	•	students.user_id → users.id

That gives you:
	•	Fast queries by school (WHERE school_id = ?)
	•	Fast queries by class (WHERE homeroom_class_id = ?)
	•	Safe 1-to-1 link between users and students
