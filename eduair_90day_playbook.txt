## The Big Engineering Questions (Think Like Uber)

Before building, you must answer these. A CTO will ask them.

**Q1: How does clock-in work when 500 students at one school all tap at 7:55 AM on Monday?**
- Problem: 500 requests hit the server at the same time
- Answer: MySQL connection pool handles concurrent requests. The `UNIQUE KEY` on `(school_id, student_id, date, shift_type)` prevents duplicate records even if two requests arrive simultaneously.
- What we need to add: Rate limiting on the clock-in endpoint

**Q2: How do we know when something breaks at 7 AM on a Monday?**
- Right now: We don't. The app crashes silently.
- What real systems do: Logging. Every request, every error, every clock-in gets written to a log file with timestamp, user_id, and what happened.
- What we need to add: Server-side logging with `winston` (Node.js library)

**Q3: How does a new school get onto EduAir?**
- Right now: You manually insert into the database
- What it should be: A school registration form — principal fills it in, submits, gets credentials
- What we need to build: School onboarding portal (Phase 2)

**Q4: How does a parent know their child is absent without opening the app?**
- Answer: Push notification (FCM) — backend sends alert to parent's phone when child is marked absent
- What we need to build: FCM integration (Phase 2)

**Q5: How do you make money?**
- See Business Model section below

## Phase 2 — Parent Portal + Notifications (May 24 – Jun 20)

**Goal: Give parents visibility. This is the feature that sells EduAir to principals.**
**Why: A principal's biggest headache is parents calling asking "where is my child?" EduAir kills that call.**

### Week 5 (May 24 – May 30) — Parent Account Flow
The `parent_students` table already exists in the database. We just need to wire it up.

**Backend tasks:**
- [ ] `POST /api/parents/:parentId/children/:studentId` — link parent to student
- [ ] `GET /api/parents/my-children` — returns list of linked students
- [ ] `GET /api/parents/child/:studentId/attendance` — returns that child's attendance

**Flutter tasks:**
- [ ] Parent shell — 2 tabs: My Children, Notifications
- [ ] My Children screen — list of linked children with today's status badge (present/absent/late)

### Week 6 (May 31 – Jun 6) — Parent Attendance Detail
- [ ] Tap a child → see their attendance calendar for this month
- [ ] Colour coded: green = present, red = absent, yellow = late
- [ ] Tap a day → see clock-in time, clock-out time, late reason

### Week 7 (Jun 7 – Jun 13) — Push Notifications
**How it works (simplified):**
1. Parent opens app → Flutter registers their device with FCM → sends device token to backend
2. Teacher marks student absent → backend checks if student has a linked parent → sends push notification to parent's device token

**Tasks:**
- [ ] Add FCM to Flutter (`firebase_messaging` package)
- [ ] Add `device_tokens` table to MySQL (stores user_id + FCM token)
- [ ] Backend: when attendance status = absent, query parent, send notification
- [ ] Test: mark a student absent → parent phone gets notification within 5 seconds

### Week 8 (Jun 14 – Jun 20) — School Registration Portal
**Why: Right now you manually add schools to the database. That doesn't scale.**

**What it needs:**
- A web form (can be built with Flutter Web or a simple HTML page)
- Principal fills in: school name, parish, MoEY school code, contact email, admin password
- Submits → record created in `schools` table + admin user created
- Principal gets login credentials by email

**Tasks:**
- [ ] `POST /api/schools/register` endpoint (public, no auth required)
- [ ] Flutter Web registration page OR simple HTML form
- [ ] Email sending with `nodemailer` (Node.js) — sends credentials to principal

**Why: Monday morning, 500 students clock in simultaneously. Your server must not crash.**

## Business Model (What You Tell a CTO or Investor)

**Problem:** Jamaican schools use paper registers. MoEYI requires SF4 compliance. Manual tracking is slow, error-prone, and untraceable.

**Solution:** EduAir — a mobile attendance system built for Jamaican schools. Digital, real-time, MoEYI-compliant.

**Revenue model (SaaS subscription):**

| School Size | Monthly Price | Annual Price |
|-------------|--------------|--------------|
| Small (under 500 students) | $5,000 JMD/month | $50,000 JMD |
| Medium (500–1000 students) | $10,000 JMD/month | $100,000 JMD |
| Large (1000+ students) | $18,000 JMD/month | $180,000 JMD |

**Why schools pay:**
- Replaces paper (saves admin time)
- Real-time parent notifications (reduces parent calls)
- MoEYI compliance built in (SF4 reporting)
- Audit trail (legal protection for the school)

**Why Microsoft/Google can't beat us:**
- They are not built for Jamaican shifts (morning/afternoon)
- They don't know MoEYI categories
- They don't integrate with JA school culture
- We do.

**5-year vision:**
- Year 1: 10 pilot schools in Jamaica
- Year 2: 50 schools, add parent paid tier
- Year 3: Caribbean expansion (Trinidad, Barbados)
- Year 4: Ministry of Education partnership
- Year 5: National infrastructure

## What a CTO Sees in 10 Minutes

When you show EduAir to a CTO, they check in this order:

1. **Does it actually work?** — Open it, log in, do something real
2. **Does the UI look professional?** — Clean layouts, no broken screens
3. **How does it handle edge cases?** — What if the student clocks in twice? What if the network drops?
4. **What does the database look like?** — Show them the ERD. Multi-tenant, audit trail, soft delete.
5. **How does it scale?** — "We use connection pooling and the unique constraint prevents race conditions"
6. **What's the business model?** — SaaS, JMD pricing, Jamaican-specific features
7. **What's next?** — Parent portal, push notifications, school registration

**Practice this pitch. 10 minutes. No notes.**
